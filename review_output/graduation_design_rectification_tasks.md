# 毕业设计整改任务清单

## 必须立即修改

### 1. 多智能体表述与代码实现不匹配

- 问题：论文和 PPT 使用 Watcher、Analyzer、Maintainer、Evaluation 四类智能体作为核心卖点，但代码中主要是 FastAPI 路由和服务函数分层，没有独立 Agent 类、调度器、消息传递或协同执行机制。
- 修改目标：避免答辩时被认为“概念写得大，代码支撑不足”。
- 建议做法：二选一处理。保守做法是把论文和 PPT 中“多智能体协同系统”改为“按智能体职责划分的流程编排”或“多角色职责分层”；如果坚持多智能体，则在代码中补充清晰的 Agent 职责类、统一调度入口、输入输出日志，并在论文第 4、5 章解释对应代码。
- 影响位置：论文 / PPT / 代码

### 2. 测试章节与本地测试代码不一致

- 问题：论文第 6 章写了单元测试、集成测试、性能测试、指标实验等，但本地 `biyesheji/tests/test_api.py` 只检查应用标题和几个页面是否能访问。
- 修改目标：让论文测试结论能被本地测试文件支撑。
- 建议做法：补充至少 5 类测试：文档文本解析、规则需求提取、需求新增修改删除后版本号变化、评估 TP/FP/FN 与 Precision/Recall/F1 计算、导出接口返回结果。论文第 6 章同步写明测试文件名、测试命令和测试结果。
- 影响位置：代码 / 论文

### 3. 线上部署缺少演示数据

- 问题：部署链接可以打开，但文档列表、需求列表、评估文档列表均为空，影响波及图返回 0 个节点和 0 条关系线。
- 修改目标：答辩时打开线上系统即可展示完整主流程。
- 建议做法：在线上环境预置一份演示需求文档、提取出的需求点、至少一次修改记录、版本链、人工基准和评估结果。答辩前逐页检查：首页统计、需求管理、追溯维护、影响波及图、AI 评估都必须有数据。
- 影响位置：部署 / 数据库 / PPT

### 4. 指标实验缺少可复现材料

- 问题：论文写了 3 份文档、人工基准 36 条、系统抽取 35 条、TP 31、FP 4、FN 5、Precision 88.57%、Recall 86.11%、F1 87.32%，但本地没有明确的实验数据和复现步骤。
- 修改目标：让指标实验可信、可检查。
- 建议做法：整理 `docs/experiment/` 或 README 小节，列出 3 份测试文档名称、人工基准表、系统抽取结果、TP/FP/FN 明细、运行步骤和截图。论文中引用这些材料。
- 影响位置：论文 / 文档 / 代码或测试数据

### 5. PPT 模板水印必须删除

- 问题：PPT 第 2 页存在“PPT下载 http://www.1ppt.com/xiazai/”字样，不适合正式答辩。
- 修改目标：保证答辩材料正式、干净。
- 建议做法：删除该水印，检查母版、页脚和隐藏文本，确保所有页面没有模板站点或无关信息。
- 影响位置：PPT

## 建议尽快修改

### 6. 统一“规则提取”和“AI”相关表述

- 问题：代码中需求点提取实际由规则完成，AI 主要用于文档整体分析和评估报告；论文/PPT 中部分地方容易被理解为 AI 直接负责需求提取。
- 修改目标：避免功能边界不清。
- 建议做法：统一改为“规则需求提取”“系统提取结果”“人工确认基准”“AI 文档分析”“AI 评估报告”。不要再写“AI 自动提取需求点”作为核心实现。
- 影响位置：论文 / PPT / README

### 7. 补充独立部署说明

- 问题：README 有运行步骤，但没有独立部署说明，也没有说明 Railway/MySQL/LLM 配置细节。
- 修改目标：让检查老师或答辩前归档人员可以复现部署。
- 建议做法：新增 `deployment.md` 或在 README 中补充：Python 版本、依赖安装、MySQL 建库、环境变量、启动命令、Railway 配置、常见错误处理、如何导入演示数据。
- 影响位置：文档 / 部署

### 8. 补充用户手册或演示脚本

- 问题：当前没有独立用户手册，答辩展示路径主要靠现场口头说明。
- 修改目标：保证答辩演示稳定。
- 建议做法：新增 `user_manual.md` 或 `demo_script.md`，按步骤写清楚：上传文档、查看需求、修改需求、查看版本链、查看影响图、建立人工基准、运行评估、导出文档。每步给出预期页面效果。
- 影响位置：文档 / PPT / 部署

### 9. 修正论文中的文件名和函数名错误

- 问题：论文中出现 `tests/test_smoke.py`，实际文件是 `tests/test_api.py`；`_init_.py` 应为 `__init__.py`；个别函数名拼写与代码不一致。
- 修改目标：论文实现章节与代码完全对应。
- 建议做法：逐项核对第 5、6 章中的文件名、接口名、函数名、数据表名，按本地代码真实名称修改。
- 影响位置：论文

### 10. 明确文档格式支持范围

- 问题：README 和 PPT 强调仅支持 docx，但上传控件和解析函数还允许 txt、md、csv、json 等文本格式。
- 修改目标：避免说明与系统行为不一致。
- 建议做法：如果答辩只展示 docx，就把页面 accept、README、论文和 PPT 都统一为“主要支持 docx”；如果保留文本格式兼容，就写清楚“docx 为主，txt/md 等纯文本格式为辅助兼容”。
- 影响位置：代码 / README / 论文 / PPT

### 11. 说明 `uploaded_docs/` 中样例文件用途

- 问题：项目中包含多份上传文档和导出文档，但没有说明哪些是演示样例，哪些是运行产物。
- 修改目标：避免归档材料显得杂乱。
- 建议做法：保留 1-3 份明确命名的演示文档，其余运行产物移到备份或不随正式归档提交；在 README 中说明样例文件用途。若这些文件依赖数据库记录展示，还要提供对应数据库 seed。
- 影响位置：代码仓库 / 文档 / 部署

### 12. 补充 AI 配置和降级说明

- 问题：AI 文档分析和 AI 评估报告依赖外部 LLM 配置，答辩现场如果 Key 或网络不可用，效果会受影响。
- 修改目标：保证答辩时能解释 AI 模块可用性。
- 建议做法：README 中说明 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL` 的配置方式；同时说明无模型配置时系统会使用本地兜底报告，现场演示前确认线上环境已配置。
- 影响位置：README / 部署 / PPT

## 可选优化

### 13. 增加数据库外键或关系说明

- 问题：当前模型主要用 `document_id`、`req_code` 等字段进行应用层关联，显式外键和 ORM relationship 不明显。
- 修改目标：提升工程规范性和论文数据库设计说服力。
- 建议做法：可以在代码中补充 ForeignKey/relationship，或至少在论文数据库设计部分说明为何采用轻量关联、如何通过应用层保证级联清理。
- 影响位置：代码 / 论文

### 14. 增加接口示例和返回样例

- 问题：README 有接口列表，但缺少典型请求和返回示例。
- 修改目标：让系统功能更容易检查和复现。
- 建议做法：为上传文档、需求修改、版本链查询、影响图查询、运行评估、文档导出各补 1 个示例。可以写在 README 或 `api_examples.md`。
- 影响位置：文档

### 15. 增加端到端截图包

- 问题：论文中有截图，但项目归档里没有单独整理截图和说明。
- 修改目标：方便答辩和归档检查。
- 建议做法：建立 `docs/screenshots/`，按页面命名保存关键截图，并在 `demo_script.md` 中引用。
- 影响位置：文档 / PPT

### 16. 强化 PPT 的结果页

- 问题：PPT 测试部分偏概括，缺少具体数据和截图串联。
- 修改目标：答辩时更容易证明“做了什么、效果如何”。
- 建议做法：增加一页“核心演示流程”和一页“实验指标结果”，放入具体 TP/FP/FN、Precision/Recall/F1、影响图截图和导出文档截图。
- 影响位置：PPT

### 17. 增加异常场景说明

- 问题：论文提到空文件、解析失败、缺少基准等异常处理，但本地测试和说明文档支撑不足。
- 修改目标：提高系统完整性。
- 建议做法：补充异常测试用例：空文件上传、非 docx 文档、文档不存在、无人工基准运行评估、导出不存在文档。测试结果写入论文或测试说明。
- 影响位置：代码 / 论文 / 文档
