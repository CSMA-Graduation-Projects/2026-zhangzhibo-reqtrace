# 毕业设计符合性检查报告

检查时间：2026-05-14

## 1. 项目总体判断

当前项目具备比较完整的毕业设计材料，代码、论文、PPT、README 和线上部署链接均已提供。代码主体不是单纯页面壳子，而是围绕“需求文档上传、规则需求提取、需求维护、来源证据、版本链、影响波及图、AI 文档分析、AI 评估、文档导出”形成了较完整的 FastAPI 原型系统；论文和 PPT 也基本围绕本系统展开。但项目仍有明显短板：多智能体表述强于代码实现，测试章节和本地测试代码不完全一致，线上部署环境没有演示数据，README 之外缺少独立部署说明、用户手册和可复现实验材料。因此当前更接近“基本框架已具备，但仍需明显补强”。

- PPT 文件：`ppt/毕业答辩.pptx`。

## 3. 项目主要优点

1. 代码结构完整，后端、模板页面、数据库模型、服务层、配置和测试目录齐全。
2. 选题特色较明确，核心功能围绕需求文档、需求变更、可追溯、版本链和评估展开，不是普通增删改查管理系统。
3. 论文第 5 章实现内容与本地代码大体对应，能看到具体文件、接口、服务函数和数据表支撑。
4. PPT 与论文主线基本一致，能概括系统背景、目标、架构、核心功能、测试和总结。
5. 线上部署可打开首页、功能页面和 Swagger UI，具备基础展示入口。

## 4. 主要问题

1. 多智能体表述偏强：论文和 PPT 使用 Watcher、Analyzer、Maintainer、Evaluation 四类智能体组织叙述，但代码中没有独立 Agent 类、调度器、消息传递或真正的多智能体协作框架，实际更像“模块职责分层 + 流程编排”。
2. 测试材料不够扎实：论文第 6 章写了较完整的单元测试、集成测试、功能测试、性能测试和指标实验，但本地 `tests/test_api.py` 只有 FastAPI 导入和几个页面访问 smoke 测试，未覆盖文档解析、规则提取、版本记录、评估指标、导出等关键逻辑。
3. 线上部署缺少演示数据：部署页面能打开，但 `/api/v1/documents`、`/api/v1/requirements/`、`/api/v1/evaluation/documents` 均返回空数组，影响波及图接口返回 0 个节点和 0 条关系线。
4. 论文/PPT 中部分术语需收敛：系统实际需求提取主要是规则提取，AI 主要用于文档分析和评估报告；应避免“AI 需求提取”“多智能体自动协同”等容易被追问的表述。
5. 说明文档不足以完全归档：README 有运行步骤和接口说明，但缺少独立用户手册、答辩演示脚本、测试数据说明、数据库初始化/演示数据导入说明。
6. README 与页面/代码有轻微不一致：README 和 PPT 强调仅支持 docx，但上传控件和解析函数也允许 txt、md、csv、json 等文本格式。
7. 示例上传文件和导出文件已放在 `uploaded_docs/`，但没有数据库 dump 或种子脚本，无法直接恢复论文中的实验状态。

## 5. 代码与系统实现检查结果

### 已有基础

- 后端采用 FastAPI，入口为 `ai_service/main.py`，已注册 `/api/v1` 接口和多个 UI 页面。
- API 路由覆盖文档、需求、变更事件、追溯版本、影响图、评估等模块。
- 数据模型覆盖 `uploaded_documents`、`requirements`、`requirement_evidences`、`change_events`、`requirement_revisions`、`document_evaluation_benchmarks`、`document_evaluation_records`。
- `document_service.py` 包含 docx 文本解析、规则需求提取、来源证据补全、需求导入、版本记录、影响波及图数据构造等逻辑。
- `document_evaluation_service.py` 包含人工基准维护、TP/FP/FN、Precision/Recall/F1 计算和评估报告生成逻辑。
- `document_export_service.py` 支持生成最新变更说明版 docx。
- 前端模板能看到需求管理、追溯维护、影响波及图和 AI 评估的交互代码，不是静态展示页。
- README 包含项目简介、功能说明、目录结构、运行步骤、环境变量和接口清单。

### 存在问题

- 本地测试代码偏弱，仅验证应用导入和页面可访问，不能支撑论文中较完整的测试结论。
- 当前检查环境未安装 FastAPI、pytest 等依赖，无法直接执行本地测试；这不一定是项目缺陷，但应在归档材料中明确依赖安装和验证命令。
- 数据库模型主要靠整数字段关联，未看到显式 ForeignKey/relationship，数据一致性更多依赖应用层清理逻辑。
- 缺少数据库初始化样例数据、演示数据导入脚本或 seed 文件，导致本地和线上难以直接复现实验截图。
- `uploaded_docs/` 中存在较多运行产生的上传和导出文件，但没有说明哪些是测试样例、哪些可删除、哪些用于论文实验。
- LLM 调用有兜底，但真实 AI 功能依赖外部 `LLM_API_KEY`、`LLM_BASE_URL`，答辩前需要确认线上配置是否可用。

### 建议补充

- 增加核心函数测试：docx 解析、规则提取、需求编号规范化、版本号递增、TP/FP/FN 计算、文档导出。
- 增加接口级集成测试：上传文档、查询需求、修改需求、查看版本链、运行评估、导出文档。
- 提供一份最小演示数据：示例需求文档、人工基准、预期抽取结果、预期评估指标。
- 补充数据库初始化或 seed 脚本，确保答辩前可一键准备演示环境。
- 清理或说明 `uploaded_docs/` 中样例文件的用途，避免被认为是无序运行产物。

## 6. 论文检查结果

### 与系统一致的部分

- 论文标题、摘要、目录和正文均围绕“需求变更管理与可追溯平台”展开。
- 论文第 3 章包含需求分析、功能需求、非功能需求和关键用例，能对应系统的文档上传、需求提取、需求维护、版本链、影响波及图、AI 评估和文档导出。
- 论文第 4 章的数据模型、页面接口和功能模块设计，与代码中的模型、路由和模板基本一致。
- 论文第 5 章提到的 `documents.py`、`requirements.py`、`trace_versions.py`、`graph.py`、`evaluation.py`、`document_service.py`、`document_evaluation_service.py`、`document_export_service.py` 等，在本地代码中均能找到。
- 论文含有系统截图、测试截图、测试表和指标实验结果，形式上比较完整。

### 可疑或偏空泛的部分

- “多智能体协同”更像论文叙述结构，代码中没有明确智能体对象、调度器或多 Agent 协作机制，容易在答辩中被追问。
- 第 6 章测试内容明显强于本地测试代码，尤其是单元测试表中提到的文档解析、兜底抽取、版本记录、评估计算、导出接口等，在 `tests/test_api.py` 中没有对应测试。
- 论文中的指标实验给出 3 份文档、36 条人工基准、35 条系统抽取、Precision 88.57%、Recall 86.11%、F1 87.32%，但本地没有配套实验数据、人工基准文件或复现实验脚本。
- 论文部分文字有小错误或不一致，例如 `tests/test_smoke.py` 与实际 `tests/test_api.py` 不一致，`ai_service/api/v1/_init_.py` 应为 `__init__.py`，部分函数名拼写与代码不一致。
- “AI 需求提取”相关表述需要统一为“规则需求提取 + AI 文档分析/AI 评估报告”，避免与代码实际逻辑冲突。

### 建议如何修改

- 将“多智能体协同”降级表述为“基于多角色职责划分的流程编排”，或补充代码层 Agent 类/调度器，否则答辩时不要夸大为完整多智能体系统。
- 在第 5 章加入更贴近代码的实现证据，例如关键接口表、关键函数输入输出、数据库表字段说明和页面请求流程。
- 在第 6 章补充测试代码截图或测试文件说明，并让本地 `tests/` 与论文测试表对应。
- 对指标实验补充样例文档名称、人工基准构建方式、系统抽取结果、TP/FP/FN 明细和复现步骤。
- 统一术语：规则提取、系统提取结果、人工确认基准、AI 文档分析、AI 评估报告。
- 修正文中实际文件名、函数名和路径错误。
